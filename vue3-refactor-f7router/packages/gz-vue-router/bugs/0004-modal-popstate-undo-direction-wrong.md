# 0004. 弹层栈内多级 popstate 被拦截时，撤销方向写反了

- **日期**：2026-07-17
- **涉及文件**：`src/router.ts`（`popstate` 监听器里 `modalIndex` 分支）

## 现象

在排查 [0003](./0003-vdialog-closeonback-conflict.md) 的过程中顺带发现：`popstate` 处理器里，
弹层栈内部多级跳转被守卫拦截时，撤销导航调用的是 `goInHistory(-1)`。

## 根因

`modalIndex` 分支只有在"目标 seq 对应弹层栈里更靠前的位置"（`modalIndex < modalStack.length - 1`）
时才会进入——这必然意味着这次 `popstate` 是一次**后退**（从更靠后的弹层退到更靠前的）。如果守卫
拦截了这次后退，正确的撤销方式是让浏览器**前进**回原来的位置（`goInHistory(1)`），而不是继续往
后退（`goInHistory(-1)`）。页面栈那边同类型的分支（`pageIndex` 分支）当时是写对的（`goInHistory(1)`），
唯独新加的弹层分支手滑复制成了 `-1`。

在 0003 那个死循环场景里，这个方向错误起到了火上浇油的作用：每次"撤销"其实是继续往后退了一层，
而不是回到原位，导致下一次 popstate 又落在一个新的、同样会被 Vuetify 守卫拦截的位置上，
循环永远找不到出口。修完 0003 的根因后这个方向错误不再会造成死循环，但只要以后任何真实业务守卫
拦截了一次弹层栈内的多级物理返回，这个方向错误依然会让"撤销"变成"越退越远"。

## 修复

```diff
- if (!allowed) goInHistory(-1);
+ if (!allowed) goInHistory(1);
```

## 教训

`goInHistory` 的撤销方向必须和"这次 popstate 本身是前进还是后退"严格反着来，不能不假思索地复制
邻近分支的写法——写的时候顺手把页面栈分支的注释抄过来看了一眼方向就假设一样了，实际上两个分支
一个总是后退（弹层栈按索引比较，命中就必然是后退）、一个是双向的（页面栈的"未找到"恢复分支要
自己判断 `goingBackward`）。类似的多分支状态机代码，方向性的字面量（`1`/`-1`）建议每次都单独
验证一遍语义，不要假设"看起来像"就是对的。
