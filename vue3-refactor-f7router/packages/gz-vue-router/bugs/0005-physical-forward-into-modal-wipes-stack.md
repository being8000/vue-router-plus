# 0005. 物理前进重新落到弹层 URL 时，把还在下面的弹层也顶掉了

- **日期**：2026-07-17
- **涉及文件**：`src/router.ts`（`popstate` 监听器里"两个栈都找不到目标 seq"的恢复分支）

## 现象

场景：设置弹层（push）打开二级弹层 → 物理返回关掉二级弹层（设置弹层还在）→ 物理前进重新打开
二级弹层。前进之后，二级弹层确实重新出现了，但**设置弹层不见了**——期望是二级弹层叠在设置弹层
上面（和最初用 `push` 打开时视觉上一样），实际却只剩二级弹层自己。

## 根因

物理前进后，`state.seq` 对应二级弹层，但它已经不在内存的 `modalStack` 里了（上一次物理返回时被
`splice` 截断掉了），落进了"两个栈都找不到目标 seq"的恢复分支。这个分支的注释写的是
"背景页/更下层弹层留着现有的栈不动，只重置弹层栈为这一层"，但代码却是：

```ts
modalStack.splice(0, modalStack.length, makeEntry(matched, state.seq!));
```

`splice(0, modalStack.length, ...)` 是**整体替换**，会把当时还在 `modalStack` 里的设置弹层一并
清掉，只留下新建的这一个 entry——代码和注释描述的行为对不上。

## 修复

```diff
- modalStack.splice(0, modalStack.length, makeEntry(matched, state.seq!));
+ modalStack.push(makeEntry(matched, state.seq!));
```

改成 `push`，不管 `modalStack` 当前是空的（刷新丢了历史，恢复出来自然只有一层）还是已经有别的
弹层在下面（这次场景），都是往上叠一层，而不是整体替换。

## 教训

写恢复/兜底分支时，注释里说的"不动"和代码里实际做的操作要对得上——这次是先写注释描述了正确的
预期行为，写代码时手滑用了 `splice(0, length, ...)`（整体替换的惯用写法，在页面栈的同类分支里
就是要整体替换），照抄到弹层分支时忘了弹层这里的语义其实是"叠加"而不是"替换"。这类"两个相邻分支
形似神不似"的地方，建议对每个分支都单独用一个能体现"已有状态是否保留"的场景验证一遍，不要只测
"从空状态恢复"这一种情况。
