# `<f7-view>` / `<f7-tab>` / `<f7-routable-modals>` 与 core Router 实例的映射关系

映射机制的关键在于：`<f7-view>` 用 **routerId + el 双 key**，`<f7-tab>` 只用 **el 单 key**，`<f7-routable-modals>` 是**全局单例，无需匹配**。

## 一、`<f7-view>` ↔ View Router 实例

映射靠两把钥匙，因为注册时机早于 DOM 挂载完成：

### 1. `routerId`（逻辑 ID，Vue 侧生成）

- [utils.js:82-85](src/vue/shared/utils.js#L82-L85) `getRouterId()`：`` `${Date.now()}_${counter}` ``，每次调用自增，`unsetRouterIds()` 在 SSR cleanup 时归零。
- [view.vue:206-219](src/vue/components/view.vue#L206-L219)：若 `setup()` 执行时全局 `f7` 已存在（App 已初始化，典型场景是非首屏的次级 `<f7-view>`），会立刻同步调用 `f7.views.create(elRef.value, { routerId, init: false, ... })`——注意此刻 `elRef.value` 还是 `null`（模板未渲染），所以先**只带 `routerId` 把 `{ routerId, instance }` push 进 `f7routers.views`**。
- 这个 `routerId` 作为 View 的 `params.routerId` 一路传进核心层，被 [router-class.js:44](src/core/modules/router/router-class.js#L44) 读作 `router.id = view.params.routerId`——即 **core Router 实例的 `id` 就是 Vue 生成的这个 routerId**。

### 2. `el`（真实 DOM 节点，挂载后补充）

- [view.vue:264-292](src/vue/components/view.vue#L264-L292)：`onMounted → f7ready` 回调里，如果上面的分支已创建过 `f7View`，此时才把 `routerData.el = elRef.value` 补上，再调用 `f7View.init(elRef.value)` 真正触发核心路由的 `mount()`/`init()`（此时 `router.el = view.el`，见 [router-class.js:97-101](src/core/modules/router/router-class.js#L97-L101)）。
- 如果 `setup()` 时 `f7` 还不存在或 `props.init=false`（典型是**首屏主视图**，App 尚未 ready），则改为在 `onMounted` 里**先 push `{ el, routerId, ... }` 到 `f7routers.views`，再调用 `f7.views.create()`**——顺序很关键：必须先注册再创建，否则 `create()` 内部同步触发的初始导航若命中组件路由，`pageComponentLoader` 查表会找不到对应项而 `reject()`。

### 3. 查找逻辑

[components-router.js:30-34](src/vue/shared/components-router.js#L30-L34)：

```js
f7routers.views.forEach((data) => {
  if ((data.el && data.el === routerEl) || (data.routerId && data.routerId === routerId)) {
    viewRouter = data;
  }
});
```

`routerEl`/`routerId` 由 [navigate.js:593-594](src/core/modules/router/navigate.js#L593-L594) 传入：`routerEl: router.el`、外层 `routerId = router.id`。两条件是 `||`，**el 匹配和 routerId 匹配任一命中即可**，这就是为什么设计上要维护两把钥匙：应对"注册时还没有 el"和"之后按 el 匹配更直接"两种时序。

> 补充：View 初始化时如果当前 URL 已经匹配到一个组件路由（即首屏就要显示的页面），并不走 `pageComponentLoader`，而是通过 [get-router-initial-component.js](src/vue/shared/get-router-initial-component.js) 直接算出 `initialPage` 塞进 `pages` 初始值，由 view.vue 自己的 render 函数渲染——这是一条不经过 `f7routers` 匹配的"快路径"，纯粹是为了首屏渲染不用等一轮异步事件握手。

## 二、`<f7-tab>` ↔ Tab Router

没有 routerId 概念，纯靠 DOM element 相等匹配：

- [tab.vue:75-96](src/vue/components/tab.vue#L75-L96)：`setup()` 阶段若 `f7` 已存在，先 push `{ setTabContent }`（还没有 `el`）；`onMounted → f7ready` 里再把 `elRef.value` 补上或重新 push 一条带 `el` 的记录。
- 核心层 [tab.js:57-65](src/core/modules/router/tab.js#L57-L65)：通过路由配置里开发者写的 **`tab.id`**（如 `{ path: '/a', tab: { id: 'tab-a', component: ... } }`）拼成 CSS 选择器 `` `#${tabRoute.id}` `` 或 `${router.view.selector} #${tabRoute.id}`，用 Dom7 `$()` 在真实 DOM 里查出 `tabEl`。
- 再调用 `router.tabComponentLoader({ tabEl, ... })`，[components-router.js:122-126](src/vue/shared/components-router.js#L122-L126) 里用这个 `tabEl` 去 `f7routers.tabs` 数组里找 `tabData.el === tabEl` 的项。

也就是说 **View 是"逻辑 ID 优先、DOM 兜底"，Tab 完全依赖开发者在路由配置里写的 `id` 属性渲染到真实 DOM 后再反查**，两者的时序保障方式相同（先注册后触发）。

## 三、`<f7-routable-modals>` ↔ Modals Router

- [routable-modals.vue:12-25](src/vue/components/routable-modals.vue#L12-L25)：`onMounted` 时直接 `f7routers.modals = routerData.value`（赋值，不是 push），`onBeforeUnmount` 时置 `null`。
- 因为一个 App 通常只挂一个这样的组件（承载所有 popup/sheet/popover/loginScreen 等 modal 路由），[components-router.js:181-188](src/vue/shared/components-router.js#L181-L188) 里直接 `const modalsRouter = f7routers.modals;` 取用，**不需要任何 key 匹配**——这是三者里最简单的映射方式，代价是"只支持全局唯一"。

## 四、汇总对比

| 组件 | 映射 key | 谁生成 key | 何时补全 | 查找方 |
|---|---|---|---|---|
| `<f7-view>` | `routerId`（逻辑）+ `el`（DOM，兜底/互补） | Vue: `getRouterId()`；Core: `router.id = view.params.routerId` | `routerId` 在创建 View 实例时就有；`el` 要等 `onMounted` | `navigate.js` 传 `router.el`/`router.id` → `components-router.js` 双条件 `||` 匹配 |
| `<f7-tab>` | `el`（DOM，唯一 key） | 无逻辑 ID，靠开发者路由配置的 `tab.id` | 必须等 `onMounted` 拿到真实 DOM | `tab.js` 用 CSS 选择器查出 `tabEl` → `components-router.js` 按 `el` 相等匹配 |
| `<f7-routable-modals>` | 无（全局单例） | — | `onMounted` 直接赋值 | `modal.js`/`components-router.js` 直接读取 `f7routers.modals` |

本质上都是同一套"**先注册进 `f7routers`，再等核心路由发起加载请求时反查**"的模式，区别只在于用什么做查找 key——View 因为要处理"实例创建早于 DOM 挂载"的时序问题，多加了一层 `routerId` 逻辑 ID 作为兜底；Tab 和 Modals 场景更简单，直接用 DOM 引用（或干脆不需要 key）。

## 五、这套设计核心解决的实际业务场景

### 1. 多视图并存（Split View / Tablet 双栏布局）

平板/桌面端常见 master-detail 布局：同一屏幕同时挂载多个 `<f7-view>`（左侧列表视图 + 右侧详情视图），每个视图各自维护独立的页面栈和浏览器历史。`f7routers.views` 用**数组**而非单例存储，正是为了支持"任意数量视图同时存在、各自独立寻址"——核心路由触发加载时不关心页面数量，只按 `el`/`routerId` 精确匹配到那一个视图实例，不会加错地方。

### 2. 底部多 Tab 应用，每个 Tab 独立路由栈

典型的"首页/消息/我的"底部 Tab 结构，每个 `<f7-tab>` 内部还可能各自嵌套一个 `<f7-view>`（tab + view 组合），互不干扰地维护浏览历史。`f7routers.tabs` 同样是数组，按 DOM 元素查找，保证 Tab 切换时核心路由能精确找到"当前要渲染内容的那个 Tab 容器"，而不会串到其他 Tab 里。

### 3. 应用启动时序不确定（首屏 View vs. 动态创建的 View）

App 首次挂载时，`Framework7` 实例、`<f7-view>` 组件、路由系统三者的初始化顺序并不固定：可能先有 App 后有 View（同步分支），也可能 View 组件先渲染但 App 还没 `ready`（异步 `f7ready` 分支）。若没有"先注册再触发"这套时序保障，直接在初始化过程中同步命中一个组件路由时，会出现"路由已经在找 View 容器，但 Vue 侧还没登记"的空窗期，导致页面加载 `reject()` 失败、白屏。这套设计通过**注册优先于触发**的强顺序，规避了框架初始化竞态问题。

### 4. 动态新增 / 销毁视图或 Tab（不写死组件树）

例如运行时按需打开一个新的独立视图窗口、动态生成 Tab 列表、条件渲染 `v-if` 控制的视图。`onBeforeUnmount` 里都会把自己从 `f7routers` 对应数组里移除——这避免了两个问题：一是核心路由后续导航时匹配到"实际已经不存在"的僵尸引用；二是数组无限增长造成的内存泄漏（这在长时间运行的 SPA/Hybrid App 里是真实会踩的坑）。

### 5. Vue 虚拟 DOM 异步渲染 与 Framework7 同步 DOM 操作 的协调

Framework7 核心路由原本假设"页面/Tab 内容加载是同步 DOM 插入"（默认实现直接 `$el.append()`）。但 Vue 组件渲染是**响应式、异步批处理**的——`setPages()` 触发的新 `<component>` 要等 Vue 下一次 flush 才会真正出现在 DOM 里。`viewRouterDidUpdate`/`tabRouterDidUpdate`/`modalsRouterDidUpdate` 这套事件握手，本质是让核心路由"等得起"——先拿到数据引用（通过映射表定位到正确的响应式数组），再等 Vue 真正渲染完成后从 DOM diff 出新增元素，才把 `pageEl` 交还给核心路由继续走动画、生命周期回调（`page:init`/`page:afterIn` 等）。没有这层映射 + 事件同步，动画时机和生命周期回调时机都会错乱。

### 6. `keepAlive` 页面复用 / Tab 内容保留

业务里常见"表单页返回后保留输入状态""Tab 切走再切回不重新加载列表"的需求。`removePage`/`removeTabContent` 里对 `keepAlive` 的特殊处理（跳过销毁，只挪 DOM）依赖能先**精确定位**到具体是哪个 View/Tab 的哪一条 page 数据——这就是为什么 `pages`/`tabContent` 要以数组形式挂在 `f7routers` 注册项上，而不是全局一份。

### 7. `openIn`：把一段路由内容临时借壳到 Popup/Sheet/Panel 里展示

业务场景如"点击列表项在弹层里预览详情，同一个路由既能整屏跳转也能弹层打开"。`openIn` 内部会动态创建一个临时的 `<f7-view>`（`$f7.view.create(viewEl, {...})`），这个临时 View 同样要走上述注册流程才能让其内部的二次导航被核心路由正确路由到——设计上的通用性使得"臨时/动态创建的视图"和"模板里声明的视图"走的是完全同一套映射机制，不需要额外分支逻辑。

### 8. 跨框架复用同一套核心路由（Vue / React / Svelte 共享 `core/modules/router`）

因为映射表和"注册-触发"协议是 Vue 适配层私有的东西，core router 本身只认 `pageComponentLoader`/`tabComponentLoader`/`modalComponentLoader` 这几个协议方法名——这让 Framework7 能用同一套路由 DSL（`routes` 配置、`tabs`、`detailRoutes`、`master` 等）无缝套在 Vue/React/Svelte 三套完全不同的渲染体系上，业务方迁移框架或多端复用路由配置时改动成本极低。

### 9. SSR 场景下多请求状态隔离

`f7routers` 是模块级单例，在 Node.js SSR 场景下同一进程会处理多个请求；`cleanup()`/`unsetRouterIds()` 在每次 `f7init` 时重置 `views`/`tabs`/`modals` 以及 routerId 计数器，避免上一个请求残留的注册项污染下一个请求的路由匹配——这是纯浏览器端 SPA 不会遇到、但服务端渲染多租户场景下必须处理的问题。
