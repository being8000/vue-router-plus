# `<f7-view>` / `<f7-tab>` / `<f7-routable-modals>` 路由跳转过渡动画设计

## 核心结论

Framework7 的过渡动画**完全由 core router 基于真实 DOM 的 CSS class + `animationend`/`transitionend` 事件驱动**，与使用哪个前端框架（Vue/React/Svelte/纯 JS）无关。Vue 适配层（`f7routers` + 事件握手机制）只负责一件事：**保证真实 DOM 节点在动画开始之前已经挂载到页面中**，之后的动画类切换、方向判断、动画结束回调全部交还给 core router 用同一套 Dom7 逻辑执行。三种容器（View 页面、Tab、Modal）分别有各自独立的动画子系统，互不复用同一份代码，但设计思路一致：**用 class 表达状态、用原生动画结束事件表达时机、用回调链驱动生命周期钩子**。

## 一、Page 级过渡动画（`<f7-view>`）

### 1. 三态 class 表达页面位置

每个 `.page` 在任意时刻只处于三种位置之一，靠 class 表达：

- `page-current`：当前显示页
- `page-previous`：位于当前页"下方"（后退目标）
- `page-next`：位于当前页"上方"（前进目标，动画时的新页起始态）

切换由 [router-class.js:581-592](src/core/modules/router/router-class.js#L581-L592) 的 `setPagePosition($el, position, ariaHidden)` 统一维护，同时处理 `aria-hidden` 无障碍属性，并触发 `page:position` 事件 / `pagePosition` router 事件。

### 2. forward（前进）动画编排 —— [navigate.js:428-501](src/core/modules/router/navigate.js#L428-L501)

顺序：

1. `pageCallback('beforeOut', $oldPage, 'current', 'previous')` / `pageCallback('beforeIn', $newPage, 'next', 'current')` —— 触发 `page:beforeout`/`page:beforein` 生命周期钩子（组件可在此做动画准备）。
2. `setPositionClasses()`：`$oldPage` 加 `page-current`（暂不变 aria-hidden），`$newPage` 加 `page-next`——此时两页同时存在于 DOM，新页处于"次态"。
3. 计算 `transition`：优先级 `options.transition` > `currentRoute.route.transition` > `currentRoute.route.options.transition` > 全局 `router.params.transition`；MD 主题下还支持 `materialPageLoadDelay`（`delay`）先 `setTimeout` 再启动动画，模拟 Material Design 的入场延迟节奏。
4. 调用 `router.animate($oldPage, $newPage, 'forward', transition, afterAnimation)`。
5. `afterAnimation()`：把 `$newPage` 置为 `page-current`，`$oldPage` 置为 `page-previous`（非 master 页会加 `aria-hidden`），触发 `page:afterout`/`page:afterin`，然后视情况移除旧页（`preloadPreviousPage`/iOS 滑动返回场景会保留旧页以便下次手势预览；`smart-select-page`/`photo-browser-page` 等特殊弹出页也会保留）。

### 3. `router.animate()` —— 动画实际执行者（[router-class.js:106-143](src/core/modules/router/router-class.js#L106-L143)）

```js
animate($oldPageEl, $newPageEl, direction, transition, callback) {
  if (router.params.animateCustom) {
    // 完全自定义动画函数，跳过默认 CSS 方案
    return router.params.animateCustom.apply(router, [$oldPageEl, $newPageEl, direction, callback]);
  }
  if (transition) {
    // 具名过渡：如 f7-cover-v / f7-dive / f7-parallax 等
    const cls = `router-transition-custom router-transition-${transition}-${direction}`;
    (direction === 'forward' ? $newPageEl : $oldPageEl).animationEnd(onDone);
    router.$el.addClass(cls);
    return;
  }
  // 默认过渡：iOS 是双页对滑，MD 是新页单向覆盖
  const routerTransitionClass = `router-transition-${direction} router-transition`;
  (direction === 'forward' ? $newPageEl : ios ? $oldPageEl : $newPageEl).animationEnd(onDone);
  router.$el.addClass(routerTransitionClass);
}
```

关键设计点：

- **动画本身是纯 CSS**（`.less` 里定义 `router-transition-forward`/`router-transition-backward` 对应的 keyframes/transform），JS 只负责加/删 class；
- **监听哪个元素的 `animationend` 决定"动画完成"的时机因主题和方向而异**：
  - forward 方向：始终监听 `$newPageEl`（新页滑入完成即算完成）；
  - backward 方向：iOS 监听 `$oldPageEl`（iOS 是旧页滑出决定耗时），MD/其他监听 `$newPageEl`（MD 是新页覆盖式滑入决定耗时）——这对应 iOS 双页联动 vs MD 新页覆盖两种设计语言的动画特性差异；
  - 有 `transition`（具名过渡）时统一走 `router-transition-custom` 分支，忽略 iOS/MD 差异；
  - `animateCustom` 优先级最高，可完全接管动画（连 class 都不加），用于业务自定义炫酷转场。

### 4. backward（后退）动画 —— [back.js:313-381](src/core/modules/router/back.js#L313-L381)

结构与 forward 对称：先 `setPositionClasses()`（`$oldPage`→`page-current`，`$newPage`→`page-previous`），再 `router.animate(..., 'backward', ...)`，完成后 `afterAnimation()` 把 `$newPage` 置 `page-current`、`$oldPage` 置 `page-next`（`aria-hidden`），随后 `removePage($oldPage)` 真正销毁旧页组件。同时会在此处处理 `preloadPreviousPage`（预加载上上级页面，配合 iOS 滑动返回手势体验）和 `History.clearRouterQueue()`（清理浏览器历史导航队列）。

### 5. 桌面宽屏 Master-Detail 场景下跳过动画

`forward`/`backward` 里都有判断：

```js
if (options.animate && !(isMaster && app.width >= router.params.masterDetailBreakpoint)) { ... }
else { afterAnimation(); } // 直接跳过动画
```

这是为了避免在平板/桌面宽屏双栏布局下，主列表页（master）切换时还播放本该用于手机单栏场景的滑动动画——业务上这是"自适应布局降级为静态切换"的关键开关。

### 6. iOS/MD 滑动返回手势（swipe-back）—— 与"动画"共用同一套结束流程

[swipe-back.js](src/core/modules/router/swipe-back.js) 实现的是**手势跟手的实时动画**，与上面"点击式导航动画"是两条独立但衔接的路径：

- `touchmove` 阶段：直接用 `transform: translate3d(...)` 实时跟随手指位置更新 `$currentPageEl`/`$previousPageEl` 的位置（不经过 CSS transition，逐帧手动计算 `percentage`），并可选实现阴影渐隐（`page-shadow-effect`）、透明度渐变（`page-opacity-effect`）两种视觉效果（由 `iosSwipeBackAnimateShadow`/`iosSwipeBackAnimateOpacity` 等参数控制）。
- `touchend` 阶段：根据滑动距离/时长判定是否触发真正的返回（`touchesDiff > 10` 且短时，或滑动过半），随后加上 `page-transitioning page-transitioning-swipeback` class 交给 CSS transition 补完剩余位移，用 `transitionEnd` 回调统一收尾——收尾逻辑与 `back.js` 里的 `afterAnimation` 几乎一致（触发 `pageCallback('afterOut'/'afterIn')`、`removePage`、更新 `history`/`browserHistory`）。

也就是说**手势驱动和点击驱动在“真正切走页面”这一步，走的是同一条 `pageCallback` + `removePage` 生命周期链**，只是动画的驱动方式（实时 transform vs. CSS class 一次性触发）不同。

### 7. Vue 层如何介入这套动画（关键：不介入 CSS/事件，只保证 DOM 就绪）

- `navigate.js`/`back.js` 在需要渲染组件页面时调用 `router.pageComponentLoader({...})`，Vue 侧实现（`components-router.js` 的 `pageComponentLoader`）把新页 push 进对应 `<f7-view>` 的响应式 `pages` 数组，触发 Vue 重渲染；
- 必须等 `viewRouterDidUpdate` 事件（由 `view.vue` 的 `onUpdated` 钩子在 Vue 真正把新 DOM 刷新到页面后触发）才 `resolve(pageEl)`，把**真实存在于文档树中的 `.page` DOM 元素**交还给 `forward()`/`backward()`；
- 只有拿到这个真实 DOM 元素，`forward()`/`backward()` 才能继续走上面第 2-6 节完全一致的 class 切换 + `animationend` 流程——**Vue 组件本身完全不知道、也不参与动画逻辑**，它只是把"要显示的组件"渲染成一个普通的 `.page` DOM 节点交出去。

这正是设计上的解耦关键：动画系统只认 DOM 和 class，不认框架；框架适配层只负责"生出 DOM"，不用关心、也不需要重新实现一遍动画。

## 二、Tab 级过渡动画（`<f7-tab>`）

Tab 切换动画由 [`src/core/components/tabs/tabs.js`](src/core/components/tabs/tabs.js) 的 `app.tab.show()` 实现，与 Page 动画是完全独立的另一套机制，按容器类型分三种情况：

### 1. `.tabs-animated-wrap` 容器 —— 横向整体平移

```js
if ($tabsEl.parent().hasClass('tabs-animated-wrap')) {
  $tabsEl.parent()[animate ? 'removeClass' : 'addClass']('not-animated');
  const transitionDuration = parseFloat($tabsEl.css('transition-duration'));
  if (animate && transitionDuration) {
    $tabsEl.transitionEnd(tabsChanged);
    animated = true;
  }
  const tabsTranslate = (app.rtl ? $newTabEl.index() : -$newTabEl.index()) * 100;
  $tabsEl.transform(`translate3d(${tabsTranslate}%,0,0)`);
}
```

所有 `.tab` 横向并排放置在 `.tabs` 容器里，通过整体 `translate3d(-N*100%, 0, 0)` 把目标 tab 移动到可视区域，靠 CSS `transition-duration` 驱动动画，`transitionend` 通知完成（`tabsChanged()` 触发 `onTabsChanged` 回调队列）。支持 RTL 镜像（`app.rtl` 时正向平移）。

### 2. `swiper-container` 容器 —— 真正的 Swiper 手势滑动

```js
if ($tabsEl[0].nodeName.toLowerCase() === 'swiper-container' && app.swiper) {
  swiper = $tabsEl[0].swiper;
  swiper.once('slideChangeTransitionEnd', () => tabsChanged());
  swiper.slideTo(newTabIndex, animate ? undefined : 0);
}
```

这种模式下 Tab 内容真正接入 Swiper 组件，同时支持"点击 tab-link 跳转"和"手势左右滑动切换"两种交互，动画完成时机由 Swiper 自身的 `slideChangeTransitionEnd` 事件决定。

### 3. 无动画容器 —— 瞬时 class 切换

以上两种容器都不满足时，只是简单 `removeClass('tab-active')`/`addClass('tab-active')`，没有过渡效果，`animated` 保持 `false`，`tabsChanged()` 会同步立即执行。

### 4. 与路由生命周期的衔接

`router.tabLoad()`（[tab.js:66-119](src/core/modules/router/tab.js#L66-L119)）调用 `app.tab.show()` 拿到 `{ $newTabEl, $oldTabEl, animated, onTabsChanged }`，内容加载完成（`onTabLoaded`）后：

```js
if ($oldTabEl && $oldTabEl.length) {
  if (animated) {
    onTabsChanged(() => {
      router.emit('routeChanged', ...);
      if (router.params.unloadTabContent) router.tabRemove($oldTabEl, $newTabEl, tabRoute);
    });
  } else {
    router.emit('routeChanged', ...);
    if (router.params.unloadTabContent) router.tabRemove(...);
  }
}
```

即：**如果本次切换有动画，必须等动画的 `onTabsChanged` 回调触发之后，才允许卸载旧 Tab 内容**（`unloadTabContent` 配置项），避免动画播放到一半内容突然消失的闪烁问题；无动画时直接同步卸载。

### 5. Vue 层的介入方式

同 Page 逻辑：`router.tabComponentLoader({...})` 由 Vue 侧实现（`components-router.js`）把内容塞进对应 `<f7-tab>` 的 `tabContent` ref，触发响应式渲染，等 `tabRouterDidUpdate` 事件确认真实 DOM 挂载后，才 `resolve(tabContentEl)` 交还给 `tabLoad()` 继续走 `app.tab.show()` 的动画流程。**Tab 动画同样完全建立在真实 DOM class 操作之上，Vue 只负责"内容何时出现"，不负责"怎么切换"。**

## 三、Modal 过渡动画（`<f7-routable-modals>`）

Modal（popup/sheet/popover/panel/loginScreen/actions/dialog 等）的动画由基类 [`modal-class.js`](src/core/components/modal/modal-class.js) 统一实现，与 Page/Tab 又是独立的第三套机制，围绕 `modal-in`/`modal-out` 两个状态 class：

### 1. `open()` —— [modal-class.js:141-169](src/core/components/modal/modal-class.js#L141-L169)

```js
function transitionEnd() {
  if ($el.hasClass('modal-out')) modal.onClosed();
  else if ($el.hasClass('modal-in')) modal.onOpened();
}
if (animate) {
  if ($backdropEl) $backdropEl.removeClass('not-animated').addClass('backdrop-in');
  $el.animationEnd(transitionEnd);
  $el.transitionEnd(transitionEnd);
  $el.removeClass('modal-out not-animated').addClass('modal-in');
  modal.onOpen();
} else {
  // 无动画：直接同步走完 onOpen + onOpened
}
```

同时监听 `animationEnd` 和 `transitionEnd` 两种原生事件（不同 modal 类型的 CSS 实现方式不同，有的用 `transition`（sheet/popover 常见），有的用 `animation`（部分主题/自定义弹出效果），双重监听保证兼容）。背景遮罩 `backdrop-in` 单独一套 class，与 modal 主体动画解耦、可分别控制。

### 2. `close()` —— [modal-class.js:174-243](src/core/components/modal/modal-class.js#L174-L243) 结构完全对称，`modal-in → modal-out`，动画结束触发 `onClosed()`。

### 3. 与路由生命周期的衔接 —— [modal.js:40-105](src/core/modules/router/modal.js#L40-L105)

`router.modalLoad()` 创建 modal 实例后：

```js
modal.on(`${modalOrPanel}Open`, () => { ...routeCallback(modal, 'open'); router.once('swipeBackMove', closeOnSwipeBack); });
modal.on(`${modalOrPanel}Close`, () => { router.off('swipeBackMove', closeOnSwipeBack); if (!modal.closeByRouter) router.back(); routeCallback(modal, 'close'); });
modal.on(`${modalOrPanel}Closed`, () => { ...nextTick(() => { router.removeModal(modal.el); modal.destroy(); }); });
modal.open(options.animate);
```

关键设计点：

- **modal 动画完全由 modal 自身的 open()/close() 负责**，router 只是监听 `Open`/`Close`/`Closed` 三个事件节点去同步路由历史（`history.push`/`back()`）和触发对应回调（`options.on.modalOpen` 等）；
- **弹层关闭若不是路由触发的（`!modal.closeByRouter`，比如用户点击 backdrop 或右滑关闭 sheet），会反向调用 `router.back()`**，把"UI 关闭"这个动作同步回路由状态，保证 URL/history 与实际显示状态一致——这是模态类路由与页面类路由最大的行为差异点：页面路由是"路由决定 DOM"，而 modal 关闭常常是"DOM 事件反向驱动路由"；
- **`swipeBackMove` 联动**：一旦用户在页面上开始 iOS 滑动返回手势（[swipe-back.js:145-146](src/core/modules/router/swipe-back.js#L145-L146) 触发 `swipebackMove`），若此时有 modal 处于打开状态会被强制 `close()`——避免"page 在滑动切换、modal 还悬浮在上面"的视觉冲突。
- modal 内容真正插入 DOM 是在 `onModalLoaded()` 里通过 `router.modalComponentLoader({...})` 完成（Vue 场景走 `f7routers.modals`），**必须等内容加载完（对 Vue 是等 `modalsRouterDidUpdate` 事件）才调用 `modal.open()`**——这保证动画开始时 modal 内部的 Vue 组件内容已经是完整渲染好的 DOM，不会出现"弹层先滑出来、内容再半秒后跳出来"的割裂体验。

## 四、三者的共同设计范式总结

| 维度 | Page（`<f7-view>`） | Tab（`<f7-tab>`） | Modal（`<f7-routable-modals>`） |
|---|---|---|---|
| 状态表达 | `page-current/previous/next` class | `tab-active` class（+ 容器整体 translate） | `modal-in/modal-out` class |
| 动画触发 | `router.$el` 加 `router-transition-*` class | `.tabs` transform 或 Swiper `slideTo` | modal 自身 `$el`/`$backdropEl` 加减 class |
| 完成时机判定 | `animationend`（按方向+主题选监听对象） | `transitionend`（wrap 模式）/ `slideChangeTransitionEnd`（swiper 模式） | `animationend` + `transitionend` 双监听 |
| 与路由生命周期衔接 | `pageCallback('beforeOut/beforeIn/afterOut/afterIn')` | `onTabsChanged()` 回调队列，决定何时 `unloadTabContent` | `modalOpen/modalClose/modalClosed` 事件，决定何时同步 history、何时 `removeModal` |
| 可否跳过动画 | 支持（`animate:false`、桌面宽屏 master 场景自动跳过） | 支持（`animate` 参数、无动画容器自动降级） | 支持（`options.animate`） |
| 可否自定义 | `animateCustom` 全接管 / `transition` 具名过渡 | 无内置具名过渡，仅动画容器类型区分 | 各 modal 类型可各自定制 CSS，无统一具名过渡概念 |
| Vue 层的角色 | 只负责让 `.page` 真实 DOM 出现（`viewRouterDidUpdate` 握手） | 只负责让 tab 内容真实 DOM 出现（`tabRouterDidUpdate` 握手） | 只负责让 modal 内容真实 DOM 出现（`modalsRouterDidUpdate` 握手） |

**一句话总结**：Framework7 把"路由要不要动画、动画播多久、播完做什么"完全交给 core router 用原生 CSS + DOM 事件实现，三种容器各自维护一套独立但同构（class 状态机 + 结束事件回调）的动画子系统；Vue（以及 React/Svelte）适配层不重新实现任何动画逻辑，只承担"让 Vue 的响应式渲染结果，在动画开始前变成一个货真价实、已挂载到文档树里的 DOM 节点"这一件事，通过 `f7routers` 注册表 + `*RouterDidUpdate` 事件握手完成这层保证。这也是为什么同一份 CSS 主题（iOS/MD）和同一套动画参数（`animate`/`transition`/`iosSwipeBack` 等）在 Vue、React、Svelte、纯 JS 四个版本里表现完全一致——动画从未属于框架层。
